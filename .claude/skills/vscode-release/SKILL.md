# /vscode-release — Cut a SemVer release

## Role
Release manager — turn the accumulated changes into a clean, versioned, marketplace-ready release.

## Goal
Stamp the pending `[Unreleased]` changelog entries into a dated SemVer version block, bump the version at its single source, and keep the canonical changelog, `package.json`, and the root mirror consistent.

## Deliverable
Updated canonical `docs/release/CHANGELOG.md` (a new released block + reset `[Unreleased]`), the bumped `package.json` `"version"`, a regenerated root `CHANGELOG.md` mirror, and a copy-pasteable release note.

---

## Prerequisite

The project must be loaded (`/vscode-load-project`, OR `docs/specs/04-architect.md` / README present). If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

Load context: `@rules/versioning.md` (+ `@rules/verification.md`).

## Step 1 — Read the current state

1. Read the canonical `docs/release/CHANGELOG.md`.
   - **Absent** → the extension predates the changelog system. Stop and route to `/vscode-load-project` (it proposes the retroactive initialization, seeding the canonical from the root `CHANGELOG.md` if one exists). Do not create it here.
   - Present → read the `## [Unreleased]` section and the current top released version.
2. Read the canonical version: `package.json` `"version"`.

## Step 2 — Nothing to release?

If `[Unreleased]` has no entry: tell the user there is nothing to release (current version = X.Y.Z) and stop. Do not create an empty version block, do not touch the root mirror.

## Step 3 — Infer and propose the increment

Apply the inference of `@rules/versioning.md`:
1. Any `**BREAKING:**` marker or a `Removed` entry → **MAJOR**.
2. Else any `Added` entry → **MINOR**.
3. Else (only `Fixed` / internal `Changed`) → **PATCH**.

Compute the three candidate versions from the current one. Ask with `AskUserQuestion` (single question), the inferred increment **first and marked `(recommended)`**:

- `MAJOR → <x+1>.0.0` — [what counts as breaking, per `@rules/versioning.md`: a command/setting removed, a contribution id renamed, the `engines.vscode` floor raised]
- `MINOR → <x>.<y+1>.0` — new backward-compatible feature(s)
- `PATCH → <x>.<y>.<z+1>` — fixes / internal changes only

Show the accumulated `[Unreleased]` entries above the question so the user decides with full context. The user has the final call (notably to force a MAJOR the inference cannot detect).

## Step 4 — Apply the release

On confirmation, using the chosen version `NEW` and the session date `<YYYY-MM-DD>`:

1. In the canonical `docs/release/CHANGELOG.md`: rename `## [Unreleased]` content into a new block `## [NEW] - <YYYY-MM-DD>` (keep the category order Added/Changed/Fixed/Removed/Security), and insert a fresh empty `## [Unreleased]` above it.
2. Bump the **canonical** version: `package.json` `"version"` → `NEW`.
3. **Regenerate the root `CHANGELOG.md` mirror** from the canonical: copy every released block (in order, newest first), **omitting** the `## [Unreleased]` section, under the mirror preamble (`# Change Log` + the Keep a Changelog / SemVer line). The mirror's top block must be the `NEW` block. This is the file that ships in the `.vsix` (`docs/**` is excluded — `@rules/manifest.md`).
4. Do **not** touch `docs/specs/`, the README, or the extension `CLAUDE.md` origin (that records the framework version, not the extension version).

## Step 5 — Verify + hand off

1. Apply `@rules/verification.md` version consistency: the top released version of `docs/release/CHANGELOG.md` == `package.json` `"version"` == the top block of the root `CHANGELOG.md` mirror; `[Unreleased]` reset empty in the canonical, absent from the mirror.
2. Output the cut version block verbatim in a fenced markdown snippet — ready to paste into a marketplace release note.
3. Remind the user that packaging, committing, tagging, and publishing are theirs to do (this skill never runs them): e.g. `npm run package` (`.vsix`), then `git commit -am "release: vNEW"` + `git tag vNEW`, then optionally `vsce publish` / `ovsx publish`.

## Anti-patterns — what NOT to do
- **Do not** create the changelog here if it is absent — route to `/vscode-load-project`.
- **Do not** cut a release with an empty `[Unreleased]`.
- **Do not** apply a MAJOR without explicit user confirmation.
- **Do not** bump `package.json` and forget to regenerate the root mirror (or vice versa) — the three locations must agree.
- **Do not** add an `[Unreleased]` section to the root mirror — it holds released versions only.
- **Do not** rewrite or reword past released blocks — only `[Unreleased]` moves; the mirror is a faithful copy of the released blocks.
- **Do not** package, commit, tag, or publish — surface the commands, let the user run them.

## When the user asks something adjacent
- **"Just bump to X.Y.Z"** → honor an explicit version, but still move `[Unreleased]` into the dated block, bump `package.json`, and regenerate the root mirror.
- **"Release and publish to the marketplace"** → do the changelog + version bump + mirror; the actual `vsce package` / `vsce publish` / `ovsx publish` stays the user's action (surface the commands).
