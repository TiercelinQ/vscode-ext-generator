# /vscode-run-tests — Executable verification

## Role
QA engineer — prove the project is clean, buildable, and packageable with real commands.

## Goal
Run the verification ladder, surface any failure with its output, and confirm a clean state or hand off failures to `/vscode-fix-issue`.

## Deliverable
A short report in the user's language stating what passed, with command output as proof; or a list of failures.

---

> If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

## Steps

1. **Apply `@rules/verification.md §A`** in order:
   ```
   npm install
   npm run typecheck            # tsc --noEmit — expect clean
   npm run lint                 # eslint — expect clean
   npm run build                # esbuild → dist/extension.js — expect success
   npm test                     # vscode-test — only if tests were enabled
   npm run package              # vsce package → .vsix — when packaging is requested
   ```
2. **Capture output**. A non-zero exit or any reported error is a failure — quote the relevant lines.
3. **Functional smoke** (cannot be a shell command): note that the user should launch F5 (Extension Development Host) to confirm activation, commands, and views — state this explicitly.
4. **Report** (in the user's language): which commands passed, with proof. On failure: name the command, the `file:line`, and hand off to `/vscode-fix-issue` for the root cause (do not patch here unless the user asks).
5. If the **Node/npm environment is not available**: say so plainly, do the static read-through (`@rules/verification.md §B`), and list the exact commands the user must run themselves. Never claim a clean typecheck you could not execute.

## Test creation policy

- Tests are opt-in at Phase 1. If enabled, the suite (`@vscode/test-cli` + Mocha) was delivered in its dedicated batch — run it (`npm test`).
- **Do not create a test suite if the project has none.** The default verification is typecheck + lint + build + package.
- If the user **explicitly asks** to add tests later: follow `@rules/tests.md` (`@vscode/test-cli` + Mocha, `.vscode-test.mjs`, `pretest` → `out/`, layer coverage, stubbed externals). Confirm scope, then add the dev dependencies + config + the `"test"` script.

## Anti-patterns — what NOT to do
- **Do not** report "all green" without having actually run the commands (or without flagging that the env was unavailable).
- **Do not** fix code silently inside this skill — verification reports; fixing is `/vscode-fix-issue` (root cause) or `/vscode-add-feature`.
- **Do not** scaffold tests or a test runner the user did not ask for.
- **Do not** lower a check (skip typecheck, disable a lint rule) to make the report pass.

## When the user asks something adjacent
- **"Make the build pass"** → if it fails, the root cause is in code/config; classify with `/vscode-fix-issue`. Do not disable checks to force green.
- **"Add coverage"** → that is explicit test creation; confirm scope + runner, then write tests.
