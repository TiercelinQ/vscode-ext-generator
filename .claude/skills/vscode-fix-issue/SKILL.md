# /vscode-fix-issue — Fix a bug (root cause)

## Role
Debugger — find the root cause and fix it cleanly, never paper over it.

## Goal
Resolve the failure so the relevant `@rules/verification.md` checks pass, with the minimum correct change.

## Deliverable
The corrected file(s) on disk + a passing verification + (after a multi-attempt anomaly) a cleanup report.

---

> If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

## Decision tree — classify the failure first

Read the actual error before touching code. Classify, then act in this order:

### 1. Typecheck / lint error (`npm run typecheck`, `npm run lint`)
- Read the exact message + `file:line`. Most are precise (type mismatch, missing import, unused symbol, `any`).
- Fix the cause. **Do not** add `// eslint-disable` or `as any` to silence it unless genuinely wrong and justified in a comment.

### 2. Build failure (`npm run build` — esbuild)
- Resolve the import/config error esbuild reports. Confirm `vscode` is `external` and the webview script (if any) is a separate entry.

### 3. Activation error (Extension Development Host)
- "command 'x' not found" / extension does not activate: trace `activate()` — an exception thrown there aborts activation. Check the `activationEvents` (auto-generated; no `"*"`), `main` → `dist/extension.js`, and that every registered command id matches a `contributes.commands` entry.
- A disposable not pushed into `context.subscriptions` leaks across reloads — fix the registration.

### 4. Command / view desync (command does nothing / view empty)
- Trace **end-to-end**: `src/constants.ts` id ↔ `contributes` (`package.json`) ↔ registered handler/provider ↔ `when`-clause ↔ caller. A mismatch in any link breaks it. Fix the broken link, keep the chain consistent.
- Empty tree: the `TreeDataProvider` returns no children or never `refresh()`es after a mutation.

### 5. State / sf issue
- State not persisted/read: check it goes through the wrapper (`storage.ts`/`secrets.ts`/`configuration.ts`) with a key from `constants.ts`. Secret in the wrong store → move to `SecretStorage`.
- sf call fails: check `sf-cli.ts` uses a `spawn` args array, the `sf` binary is detected, and the envelope is parsed defensively (`@rules/sf-cli.md`). If a wrong command/flag is suspected, verify it against the matching `sf-cli-reference/` section file.

### 6. Security gap (`@rules/security.md`)
- Webview without CSP+nonce, remote resource, unvalidated message/arg, secret outside `SecretStorage`, `shell:true`/concatenated `sf` command. These are never "acceptable for now" — fix to the rule.

### 7. Webview / theming deviation (visual bug)
- Hardcoded color/size, a `@media (prefers-color-scheme)`, a non-themed element. Route the value through a `--vscode-*` token and the theme classes. See `webview-ui.md`.

## Steps

Read `webview-ui.md` on demand if the fix touches webview UI (not auto-imported). If the fix touches `sf` calls (an `sf-cli.ts` helper, a flag, a subcommand), read `@rules/sf-cli.md` and consult `sf-cli-reference/INDEX.md` → the matching section file to verify the exact command/flag (never read the whole catalog, never guess a flag).

1. Get the real error (command output, Dev Host console, or the user's repro). If you cannot reproduce, ask for the exact message — do not guess.
2. Classify with the tree above; read both the failing site and the declaration it depends on.
3. **Ask: "knowing what I know now, what is the clean fix?"** Implement that, not the quickest patch. If the clean fix is larger than expected, say so before applying.
4. Apply the minimum correct change. Respect the layers, the contract (`docs/specs/04-architect.md`), and the security rules.
5. **Verify**: re-run `@rules/verification.md §A` for the affected area (and the F5 smoke if relevant); confirm the targeted failure is gone and nothing else broke. Then apply `@rules/readme.md` — if the fix changed a README-documented aspect (a contribution point, a dependency, a prerequisite), regenerate the README.
6. If it took several attempts: produce the **cleanup report** (`@rules/architecture.md` — list every dead element added during failed attempts), then offer `Do you want to remember this point? /vscode-save-memory`.

## Anti-patterns — what NOT to do
- **Do not** silence a typecheck/lint finding with `as any` / `// eslint-disable` instead of fixing the cause.
- **Do not** weaken security to make an error disappear — fix the actual cause within `@rules/security.md`.
- **Do not** wrap a bug in a `try/catch {}` that hides it.
- **Do not** hardcode a value to "make it look right" in a webview — fix it through the `--vscode-*` tokens.
- **Do not** patch only one link of a desynced command/view chain and leave it inconsistent.
- **Do not** leave failed-attempt residue in the file. Clean it up.
- **Do not** expand the fix into an unrequested refactor; flag genuinely-wrong surrounding code for `/vscode-refactor-code`.
- **Do not** claim it is fixed without re-running the verification (or telling the user which command to run if no env).

## When the user asks something adjacent
- **"Just make the error go away"** → the error is a symptom; explain you are fixing the cause. A genuine temporary silence is explicit and flagged as debt.
- **"It's slow on activation"** → check for eager work in `activate()` / a `"*"` activation event; lazy-load and rely on the contributed activation events.
