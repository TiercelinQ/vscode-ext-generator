# /vscode-refactor-code — Restructure without behavior change

## Role
Cautious refactorer — improve structure, preserve behavior, never act without sign-off.

## Goal
Propose a clear restructuring plan, get explicit validation, then apply it with no behavior change.

## Deliverable
A validated plan, then the refactored files on disk + a passing verification + an updated `docs/specs/04-architect.md` if the structure changed.

---

## Hard rule

**No refactor without an explicitly validated plan.** This skill never edits code on first contact. It signals → proposes a plan → waits → only then applies. (Aligned with the global behavior rules.)

> If the project root has not been provided in this flow, first ask: `Project root? (folder path)`.

## Steps

1. **Load context**: `docs/specs/04-architect.md`, then `@rules/architecture.md` · `@rules/security.md` · `@rules/state.md` (not auto-imported); `webview-ui.md` if webview UI is involved; `@rules/sf-cli.md` if the refactor touches `sf-cli.ts` / the sf controllers (consult `sf-cli-reference/` by section to keep every command/flag intact — a refactor preserves behavior, so no command/flag may drift).

2. **Diagnose** what is actually wrong: duplication, a view doing business logic, a god-controller, a model creating UI, a hardcoded id bypassing `constants.ts`, a state access bypassing the wrappers, a hardcoded value bypassing a `--vscode-*` token in a webview. Anchor each finding to `file:line`.

3. **Decide per finding — factorize OR leave:**
   - **Factorize** when the blocks express the **same intent** (same operation, same data shape), are non-trivial (> ~5 meaningful lines of real logic), and a natural home exists (a shared model helper, a shared type in `types.ts`, a constant in `constants.ts`) **without** forcing a long bag of options or a `mode`/`kind` flag to fork behavior.
   - **Leave** when the blocks are coincidentally similar but conceptually independent, or when the shape will diverge soon. When in doubt, leave it.

4. **Propose the plan** (in the user's language): the findings, the proposed change per finding (factorize/leave + where the helper lives), the files touched, and the explicit promise of **zero behavior change** and **zero security regression**. Wait for validation.

5. **Apply** only after validation. Minimum diff. Respect the layers, the contract, and `@rules/security.md`.

6. **Verify**: `@rules/verification.md §A` — behavior unchanged, typecheck/lint/build clean. If the structure changed (new shared file, moved code, renamed command/view id), update `docs/specs/04-architect.md`, regenerate the README (`@rules/readme.md`), and keep the constants ↔ contributes ↔ handlers chain consistent end-to-end.

## Anti-patterns — what NOT to do
- **Do not** refactor without a validated plan, ever.
- **Do not** factorize two blocks into a helper with a `mode`/`type` flag that forks inside — that is two functions sharing a body.
- **Do not** create a shared abstraction "for the future" before a second real caller exists.
- **Do not** change behavior under the cover of a refactor — if behavior must change, that is `/vscode-add-feature`, declared separately.
- **Do not** weaken security while "cleaning up" (broadening the webview surface, loosening validation, widening `localResourceRoots`).
- **Do not** move an id out of `constants.ts` or a value out of the `--vscode-*` tokens into a local constant — refactors keep the centralization.
- **Do not** touch pre-existing dead code unless the user asked to remove it.

## When the user asks something adjacent
- **"Just clean it up however you think is best"** → still propose the plan first; "best" is a judgment the user should confirm before files move.
- **"Refactor and add X"** → split: the refactor (behavior-preserving, validated) and the feature X (`/vscode-add-feature`) are separate steps with separate validation.
