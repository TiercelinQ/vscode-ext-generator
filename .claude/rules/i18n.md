# Internationalization rules — VS Code extension (FR/EN)

Conditional — enabled if Phase 1 i18n = `Yes`. FR is the default. VS Code localizes in **two distinct places**: the manifest (static `contributes` strings) and the runtime code (messages, labels). Both are required.

## 1. Manifest strings — `package.nls.json`

Strings in `package.json` (`displayName`, `description`, command `title`, view `name`, config `description`) use `%key%` placeholders resolved from NLS files at the project root:

```jsonc
// package.json
{ "displayName": "%extension.displayName%",
  "contributes": { "commands": [{ "command": "myext.entity.refresh", "title": "%cmd.entity.refresh%" }] } }
```

```jsonc
// package.nls.json            (default — English fallback)
{ "extension.displayName": "My Extension", "cmd.entity.refresh": "Refresh" }
// package.nls.fr.json         (French)
{ "extension.displayName": "Mon extension", "cmd.entity.refresh": "Rafraîchir" }
```

- `package.nls.json` is the fallback (used when the active display language has no file). With FR as the product default, keep the user-facing wording consistent, but the fallback file is conventionally English; ship `package.nls.fr.json` for French. VS Code picks the file matching the editor display language.

## 2. Runtime strings — `vscode.l10n`

Code-side messages (notifications, quick pick labels, tree labels) go through `vscode.l10n.t`. Declare `"l10n": "./l10n"` in `package.json`; ship a bundle per language:

```ts
import * as vscode from "vscode";
vscode.window.showInformationMessage(vscode.l10n.t("Org {0} set as default", alias));
```

```jsonc
// l10n/bundle.l10n.json        (default keys = source strings)
{ "Org {0} set as default": "Org {0} set as default" }
// l10n/bundle.l10n.fr.json     (French)
{ "Org {0} set as default": "Org {0} définie par défaut" }
```

- `vscode.l10n.t` resolves against the bundle matching `vscode.env.language`. Use `{0}`, `{1}` placeholders — never string-concatenate a translated fragment.
- **Generate / verify the default bundle with the official extractor** instead of hand-maintaining it: `npx @vscode/l10n-dev export ./src` scans every `vscode.l10n.t(...)` call and writes `bundle.l10n.json` (keys = the source strings). Run it after adding/removing a runtime string to keep the bundle in sync; translators copy it to `bundle.l10n.<lang>.json`. VS Code falls back to the **source string** when no language bundle matches, so the identity-mapped default bundle is technically optional — ship it anyway for consistency and to seed the translation files.
- Webview strings: translate **host-side** with `vscode.l10n.t` in the HTML builder (or post translated text via the message bridge) — never hardcode UI text in the webview script. Set the document language from the editor too: `<html lang="${vscode.env.language}">`.

## 3. Rules

- Zero hardcoded user-facing string in commands/notifications/tree labels — `vscode.l10n.t` (runtime) or `%key%` (manifest).
- Dotted/descriptive keys; keep the manifest NLS keys and the runtime bundle separate (they are different mechanisms).
- **`package.nls.json` (the fallback) must be complete**: every `%key%` referenced in `package.json` needs an entry there. `vsce package` / `vsce ls` **fail** with `ERROR No translation found for %key%` on a missing key — a free automated check of the manifest i18n wiring (the runtime `l10n` bundle is not validated by vsce, only the manifest NLS).
- If i18n **not** enabled: write strings in plain FR directly (default product language), no NLS/l10n scaffolding. Keep them centralized enough that enabling i18n later is cheap, but do not pre-build the bundles.

## Anti-patterns — what NOT to do

- **Do not** localize only the manifest or only the runtime — both `package.nls*.json` and `l10n/bundle.l10n*.json` are needed when i18n is on.
- **Do not** concatenate translated fragments — use `{0}` placeholders in `l10n.t`.
- **Do not** hardcode UI text inside the webview script — pass translated strings from the host.
- **Do not** scaffold NLS/l10n files when i18n is off.

## Integrity verification

Detailed in `@rules/verification.md`. Key points (if i18n = Yes): both layers localized — `package.nls*.json` for the manifest and `l10n/bundle.l10n*.json` for the runtime (`vscode.l10n`); every key used, none missing; no concatenated translated fragment (`{0}` placeholders only); webview strings passed in translated from the host, never hardcoded in the webview script. If i18n = No: no NLS/l10n file scaffolded.
